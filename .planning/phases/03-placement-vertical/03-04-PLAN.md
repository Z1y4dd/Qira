<objective>
Build the entire kid-facing placement UI: the route group (`(authenticated)/placement/`), the start screen, the question loop (passage screen + question screen with 4 choice cards), the result screen, and the Server Actions that wire them to the Plan 03 service layer. The choice-submit pattern uses one hidden `<form>` per choice card with a bound Server Action — the chosen choice ID is baked in at render time (server-side), never travels from client to server as user input. This is the load-bearing PLAC-04 mitigation. The EscapeHatch component shell (Plan 01 Wave 0) is embedded on PassageScreen + QuestionScreen + the result page in this plan; the abort-action wiring inside that component is filled by Plan 05 in wave 4.

Purpose: PLAC-01 (the start screen + the question loop the gate from Plan 06 routes to), PLAC-02 (renders the placeholder bank from Plan 02), PLAC-04 (server-authoritative scoring — the bundle-leak invariant from Plan 01 + the form-per-choice render pattern), PLAC-05 (result screen with "we picked Level X for you"), PLAC-06 (escape hatch visible during AND **after** placement — the result page embed is what delivers SC4), PLAC-08 (the attempts row write happens inside the service called from the actions here).

Output: The complete kid-facing placement flow runs locally — `pnpm dev`, sign in as a parent, pick a child, navigate to `/placement/start`, complete 15 questions tapping choice cards, land on the result screen with "اخترنا لك المستوى ٤" AND see the two escape-hatch buttons on the result page. Bundle-leak invariant test stays green. The Playwright `placement-escape-hatch.spec.ts` flow-related cases land green after Plan 05 (action wiring), and `placement-flow.spec.ts` cases land green after Plan 06.
</objective>
