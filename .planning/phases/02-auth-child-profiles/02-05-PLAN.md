# Plan 05: Active-Child Picker + Nav Chip

## Goal

`/choose-child` is the always-shown picker after sign-in. Selecting a child sets the `qira_active_child` HTTP-only cookie via a Server Action and redirects to `/dashboard`. The avatar chip in the authenticated layout's top-right (top-left in RTL) shows the active child's display name + grade band; tapping opens a menu listing other profiles + "إدارة الملفات" (manage profiles) link. Switching is another Server Action that resets the cookie and reloads. `requireActiveChild()` Service body is the foundation Phase 3 placement and Phase 4 reader will compose against.

## Demo gate

`pnpm dev`, sign in → automatically lands at `/choose-child` showing all of the parent's children as cards + an "إضافة طفل" card. Click a child → cookie set → `/dashboard` renders with the avatar chip in the nav showing the picked child's name. Click chip → menu opens with the other children + "إدارة الملفات". Click another child → cookie reset → page reloads showing the new active child. Visit `/dashboard` without the cookie → redirected to `/choose-child`.

## Requirements covered

- **PROF-04** — Profile picker shown on entry; sets signed (RLS-trust-boundary) active-child cookie.
- **PROF-05** — Switching active child is a Server Action; resets cookie with no client-side leak.

- **Task 5.1 — `src/lib/active-child-cookie.ts` helpers**
  - **Files:** `src/lib/active-child-cookie.ts`
  - **Action:**
    - Three pure helpers (NO `next/*` imports — this file is consumed by both Server Actions and the Service Layer):
      - `export const ACTIVE_CHILD_COOKIE = 'qira_active_child' as const;`
      - `export function activeChildCookieOptions(): { httpOnly: boolean; secure: boolean; sameSite: 'lax'; path: string }` — returns `{ httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'lax', path: '/' }`.
      - `export function parseActiveChildCookie(raw: string | undefined): string | null` — returns the value if it's a syntactically valid UUID v4 (regex check), otherwise null. UUID validation prevents downstream SQL injection vectors via the cookie.
  - **Verification:** `pnpm tsc --noEmit` clean. `tests/invariants/service-layer-purity.test.ts` still green (no `next/*` imports).
  - **Commit:** `feat(lib): active-child cookie helpers (name, options, UUID parser)`

- **Task 5.2 — `requireActiveChild()` Service body**
  - **Files:** `src/services/profiles.ts`
  - **Action:**
    - Replace stub. Signature: `export async function requireActiveChild(supabase: SupabaseClient, activeChildCookieValue: string | undefined): Promise<ChildProfile>` — the cookie value is injected by the caller (Server Action / layout) to keep the Service Layer pure.
    - Body:
      1. `const parent = await requireParent(supabase);`
      2. `const childId = parseActiveChildCookie(activeChildCookieValue);` (using the helper from Task 5.1)
      3. If `childId === null` → throw `new AuthError('NO_ACTIVE_CHILD')`.
      4. `const [row] = await db.select().from(childProfiles).where(and(eq(childProfiles.id, childId as ChildId), eq(childProfiles.parentId, parent.parentId))).limit(1);`
      5. If `row === undefined` → throw `new AuthError('NO_ACTIVE_CHILD')` (cookie is stale or another parent's child).
      6. Return mapped `ChildProfile`.
  - **Verification:** `pnpm tsc --noEmit` clean.
  - **Commit:** `feat(services/profiles): real requireActiveChild body with cookie-bound child lookup`

- **Task 5.4 — Modify `(authenticated)/layout.tsx` to route to `/choose-child` when no active child**
  - **Files:** `src/app/(authenticated)/layout.tsx`
  - **Action:**
    - After the `requireParent` block (from Slice 1), add a second block:
      ```ts
      const path = (await headers()).get('x-pathname') ?? ''; // optional: use Next.js dynamic params or a sentinel header
      const isChooseChildPage = path === '/choose-child' || path.startsWith('/choose-child');
      if (!isChooseChildPage) {
        const cookieStore = await cookies();
        const raw = cookieStore.get(ACTIVE_CHILD_COOKIE)?.value;
        try {
          await requireActiveChild(supabase, raw);
        } catch (err) {
          if (err instanceof AuthError && err.reason === 'NO_ACTIVE_CHILD') redirect('/choose-child');
          throw err;
        }
      }
      ```
    - **Note on `x-pathname`:** Next.js does not expose the current route to a layout via a header by default. Two clean alternatives: (1) skip the path check entirely and instead make `/choose-child` its own route group `(authenticated)/(picker)/choose-child/page.tsx` with a sibling layout that intentionally omits the `requireActiveChild` block (cleanest), OR (2) add a custom header `x-pathname` in middleware (`src/proxy.ts` already runs on every authenticated route). **Pick (1).**
    - Refactor: create `src/app/(authenticated)/(picker)/layout.tsx` that only enforces `requireParent` (no active-child check). Move `/choose-child` and `/profiles/new` under `(picker)`. Other authenticated routes stay in the existing layout which enforces both.
  - **Verification:** Visit `/dashboard` with no cookie → redirected to `/choose-child`. Visit `/choose-child` with no cookie → renders fine.
  - **Commit:** `feat(authed): split (authenticated) into (picker) (no-active-child-needed) and default subtree (active-child required)`
